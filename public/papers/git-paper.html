Git, do zero

Controle de versão pra quem ouve falar de commit, push, branch e finge entender. Construímos o entendimento de baixo pra cima — modelo mental antes dos comandos.

KR
por Kruta
25 min de leitura · 8 capítulos

§ 01A história

por que Git existe?

Imagina que você está escrevendo um TCC. Abre o Word, digita 30 páginas, salva como tcc.docx. No dia seguinte quer mexer, mas tem medo de estragar o que já está pronto. O que você faz?

Salva como tcc_v2.docx.

Depois tcc_v2_final.docx.

Depois tcc_v2_final_DESSA_VEZ_VAI.docx.

Depois tcc_v2_final_correcao_orientador.docx.

Familiar? Esse é o problema que controle de versão resolve. Em vez de manter cópias com nomes cada vez mais desesperados, você usa uma ferramenta que guarda o histórico de tudo que você fez e te deixa voltar pra qualquer ponto anterior quando quiser.

Agora multiplica isso por um time inteiro de programadores mexendo nos mesmos arquivos ao mesmo tempo. Sem controle de versão, é um inferno. Com controle de versão, é só uma quarta-feira normal.

Entra o Linus Torvalds

Em 2005, o Linus Torvalds (sim, o cara do Linux) tinha um problema. O kernel do Linux era desenvolvido com uma ferramenta chamada BitKeeper, que era proprietária mas dava licença grátis pra projetos open-source. Teve uma briga política, a BitKeeper revogou a licença, e o Linus ficou sem ferramenta pra gerenciar o maior projeto open-source do mundo.

Ele tinha duas opções: usar CVS ou SVN (que ele detestava), ou escrever a própria ferramenta. Adivinha qual ele escolheu.

Em duas semanas, ele escreveu a primeira versão do Git. Os requisitos eram: ser absurdamente rápido, ter design distribuído (sem servidor central), suportar milhares de branches em paralelo, aguentar projetos enormes e garantir integridade — impossível alterar o histórico sem deixar rastro.

Hoje, Git é o padrão da indústria. Praticamente todo código que roda no mundo passa por Git em algum momento.

!
git ≠ github

Git é a ferramenta de controle de versão — roda no seu computador, offline, criada pelo Linus. GitHub é uma empresa (hoje da Microsoft) que oferece hospedagem de repositórios Git na nuvem. Git é o motor. GitHub é uma garagem na nuvem onde você estaciona seus projetos.

§ 02O modelo mental

como o Git pensa

Antes dos comandos. Antes do terminal. Você precisa entender como o Git enxerga o mundo. Se pular essa parte, vai decorar comandos sem entender. Se ler com atenção, o resto vira tranquilo.

Snapshots, não diferenças

Outras ferramentas de versionamento (tipo SVN) guardam o histórico como uma lista de diferenças entre versões. Tipo: "na versão 2, a linha 47 mudou de X pra Y". Pra reconstruir, ela aplica essas diferenças por cima da versão anterior.

O Git faz diferente. Cada vez que você "salva", ele tira uma foto (snapshot) de todos os arquivos do projeto naquele momento. Guarda o estado completo, não as diferenças.

"Mas isso não ocupa muito espaço?" Não, porque o Git é esperto. Se um arquivo não mudou entre dois snapshots, ele só guarda um ponteiro pro arquivo anterior. Esse modelo é o que torna o Git absurdamente rápido em operações como ver histórico, comparar versões, ou voltar pra um ponto no passado.

As três áreas do Git

Quando você trabalha com Git, seus arquivos vivem em três lugares conceituais. Essa é a parte mais importante desse paper inteiro. Se você entender isso, entendeu Git.

  • Working Directory — a pasta onde você está editando. É o que você vê no Finder ou VS Code.
  • Staging Area (Index) — uma área intermediária onde você "monta" o próximo commit, como uma sacola antes do caixa.
  • Repository — o histórico permanente. Quando você commita, o que estava na Staging vira um snapshot oficial.

Por que a Staging existe?

Outras ferramentas não têm Staging — você muda o arquivo e commita. Por que o Git complicou? Resposta: controle.

Imagina que você editou 5 arquivos. 3 deles são uma feature de login. 2 são correção de um bug no header. Você quer fazer dois commits separados: um pra cada coisa. Com Staging, você adiciona os 3 primeiros, commita, depois adiciona os outros 2, commita de novo. Limpo. A Staging é onde você monta o commit.

§ 03Mãos na massa

comandos fundamentais

Agora sim. Cada comando vai fazer alguma coisa em uma dessas três áreas. Se você sempre pensar "onde esse comando atua?", nunca mais se perde.

Configuração inicial (faz uma vez na vida)

git config --global user.name "Seu Nome"
git config --global user.email "seu@email.com"

Isso diz pro Git quem você é. Todo commit vai gravar seu nome e email — é assim que se sabe quem fez o quê.

git init — criando um repositório

Você tem uma pasta com um projeto e quer começar a versionar. Entra na pasta pelo terminal e:

git init

Isso cria uma pasta escondida .git dentro do seu projeto. Toda a mágica do Git mora nessa pasta. Histórico, configurações, branches — tudo. Se você apagar a .git, perde o histórico (mas os arquivos do projeto continuam ali).

git status — seu melhor amigo

Esse comando você vai usar mais que qualquer outro. Ele mostra em que estado as três áreas estão: o que está modificado, o que está em staging, o que ainda não foi rastreado. Sempre que estiver perdido, git status. Sempre.

git add, git commit

git add <arquivo> manda mudanças do Working Directory pra Staging. git commit -m "mensagem" pega tudo que está na Staging e congela num snapshot permanente no histórico.

O ciclo básico

# 1. edita arquivos no Working Directory
# 2. vê o que mudou
git status

# 3. manda pra Staging
git add arquivo1 arquivo2

# 4. salva no Repository
git commit -m "mensagem descritiva"

# 5. repete

Esse é o loop fundamental do Git. Tudo mais é variação ou extensão disso.

✓
boas mensagens de commit

Comece com verbo no imperativo: "Adiciona", "Corrige", "Refatora". Específico mas conciso. "Mudei umas coisas" é uma mensagem péssima. "Corrige validação no formulário de cadastro" é boa.

§ 04Branches

a parte que muda tudo

Até aqui você tem uma linha do tempo linear: commit 1, commit 2, commit 3. Bonitinho. Mas a vida real não é linear.

Imagina que você está num projeto, tudo funcionando. Aí quer experimentar uma feature nova — tipo um botão de modo escuro. Não sabe se vai dar certo. Se der ruim, não quer ter que desfazer um monte de coisa. Como faz?

Branches. Uma branch é uma linha de desenvolvimento paralela. É como dizer: "a partir desse commit, vou começar a fazer experimentos numa realidade alternativa. Se der certo, eu junto com a principal. Se der errado, eu jogo fora e a principal nem percebe."

É a razão pela qual Git domina o mundo. Outras ferramentas tinham branches, mas eram pesadas, lentas, dolorosas. No Git, criar uma branch é instantâneo.

Por que isso é genial

Porque permite que várias pessoas trabalhem em várias coisas ao mesmo tempo sem se atrapalhar. Cada feature, cada bugfix, cada experimento vive numa branch própria. Tá pronta? Vai pra main. Não tá? Fica lá, isolada.

HEAD — o "você está aqui"

Tem um conceito que aparece o tempo todo: HEAD. É um ponteiro que aponta pra onde você está agora. Geralmente aponta pra última commit da branch atual. Quando você muda de branch, HEAD vai junto.

Conflitos

Quando você dá merge, o Git tenta juntar as duas linhas do tempo. Na maior parte das vezes, dá certo automaticamente. Às vezes, dá conflito — quando as duas branches modificaram a mesma linha do mesmo arquivo de jeitos diferentes. Aí o Git para e te pede pra resolver na mão. Chato, mas faz parte. Você abre o arquivo, escolhe qual versão fica, git add, git commit.

§ 05Remoto

o mundo lá fora

Tudo que a gente fez até aqui é local — só no seu computador. Ninguém vê. Pra compartilhar, ter backup, colaborar, você precisa de um repositório remoto.

O remoto geralmente fica num serviço tipo GitHub, GitLab ou Bitbucket. É uma cópia do seu repositório que mora num servidor.

Conectando local e remoto

git remote add origin https://github.com/voce/seu-repo.git

origin é o apelido do remoto. Por convenção, o principal sempre se chama origin. Poderia ser "github" ou "carlinhos", mas todo mundo usa origin, então usa também.

git push e git pull

git push manda commits locais pro remoto. git pull traz commits do remoto pro local. Os dois são o coração do trabalho em equipe.

O ciclo completo do dia a dia

git pull origin main           # puxa atualizações do time
git checkout -b minha-feature  # cria branch nova
# ...edita arquivos...
git add .
git commit -m "mensagem"
git push origin minha-feature  # manda sua branch
# abre Pull Request no GitHub
# time revisa, aprova, faz merge na main
git checkout main
git pull origin main           # atualiza main local
# recomeça pra próxima feature

Esse é o fluxo padrão do desenvolvimento profissional. Decora.

i
.gitignore

Tem arquivos que não devem ir pro repositório: senhas, node_modules/, arquivos compilados, configs pessoais do editor. Cria um arquivo .gitignore na raiz listando os padrões a ignorar e o Git finge que esses arquivos não existem.

§ 06Quando as coisas dão errado

comandos pra desfazer

Você vai quebrar coisas no Git. Todo mundo quebra. Abaixo, os comandos pra desfazer.

"Editei errado, quero voltar o arquivo"

git restore arquivo.txt        # versão moderna
git checkout -- arquivo.txt    # versão antiga
!
cuidado

Isso apaga suas mudanças não commitadas. Não tem volta.

"Adicionei errado à Staging, quero tirar"

git restore --staged arquivo.txt

Tira da Staging mas mantém a mudança no Working Directory. Seguro.

"Commitei errado, quero desfazer o último commit"

# desfaz o commit mas mantém as mudanças
git reset --soft HEAD~1

# PERIGOSO: desfaz E joga fora as mudanças
git reset --hard HEAD~1

"Tô completamente perdido"

Existe um comando salvador: git reflog. Ele mostra tudo que o HEAD apontou recentemente, mesmo coisas que você "perdeu". Quase nada se perde de verdade no Git — quase sempre dá pra recuperar via reflog. Decora esse nome: reflog é seu paraquedas.

§ 07Por baixo dos panos

o que rola por dentro

Você já consegue trabalhar com Git agora. Mas pra virar um usuário fluente, vale entender o que está rolando por baixo.

Por baixo, o Git é um banco de dados chave-valor de objetos:

  • Blob — conteúdo de um arquivo.
  • Tree — uma pasta (lista de blobs e outras trees).
  • Commit — um snapshot (aponta pra uma tree raiz, mais metadados).
  • Tag — uma referência fixa a um commit específico.

Cada objeto é identificado por um hash SHA-1 — aquele código tipo a3f5e9c8d2… que você vê o tempo todo. O hash é gerado a partir do conteúdo. Se o conteúdo muda, o hash muda. É por isso que é impossível alterar o histórico sem deixar rastro — qualquer mudança propaga em cadeia.

Uma branch é só um ponteiro

O que é uma branch? É literalmente um arquivo de texto com o hash do commit pra onde ela aponta. Sério. Você pode ver:

cat .git/refs/heads/main
# a3f5e9c8d2b1e4f7c0a9d8b6e5f4a3c2d1e0f9a8

Aparece um hash. Só isso. Uma branch é só um nome amigável pra um hash. Por isso criar branch é instantâneo — você só está criando um arquivinho de texto com 40 caracteres.

§ 08Para onde ir daqui

próximos passos

Você cobriu o suficiente pra trabalhar profissionalmente. Git é fundo, e tem coisas legais que ficam pra depois:

  • git rebase — alternativa ao merge que reescreve o histórico.
  • git cherry-pick — pegar um commit específico de outra branch.
  • git stash — guardar mudanças temporariamente sem commitar.
  • git tag — marcar versões importantes (tipo v1.0.0).
  • Hooks — scripts que rodam em eventos (pre-commit, pre-push).
  • git bisect — busca binária pra encontrar o commit que introduziu um bug.

Cada um dá pra explorar quando precisar. Não tenta aprender tudo de uma vez — você vai esquecer e ficar frustrado.

Pra estudar mais

  • Pro Git Book (git-scm.com/book/pt-br/v2) — gratuito, traduzido, oficial. A bíblia.
  • Atlassian Git Tutorials — explicações visuais excelentes.
  • Oh Shit, Git!?! (ohshitgit.com) — receita de bolo pra quando você quebra alguma coisa.
  • Learn Git Branching — referência interativa.
· · ·

Git não é difícil. Git é mal explicado. A maior parte dos tutoriais joga comandos na sua cara sem te dar o modelo mental que faz eles fazerem sentido. Se você chegou até aqui, você tem o modelo mental: três áreas, snapshots como objetos, branches como ponteiros, HEAD como o "você está aqui".

A partir disso, tudo no Git fica navegável. Você pode esquecer um comando — mas sabe procurar. Pode ver um erro estranho — mas sabe interpretar. Pode encarar um merge conflict cabeludo — mas sabe o que está fazendo.
